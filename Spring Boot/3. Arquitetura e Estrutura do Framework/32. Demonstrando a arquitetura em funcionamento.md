
# 1. Conceito Fundamental

No JPA, o ID define a identidade da entidade.

O Hibernate usa o valor do ID para decidir:

- Se deve executar INSERT
    
- Ou se deve executar UPDATE
    

Regra prática do Spring Data:

- ID == null → INSERT
    
- ID != null → UPDATE (merge)
    

O ID é o que determina se a entidade já existe no banco.

---

# 2. O Método PUT e o Problema do ID

Trecho do código:

@PutMapping("{id}")  
public void atualizar(@RequestBody TodoEntity todo, @PathVariable("id") Integer id) {  
    todo.setId(id);  
    this.service.atualizar(todo);  
}

Observe:

- O ID vem na URL
    
- O corpo da requisição pode não conter o ID
    
- O objeto desserializado pelo Jackson geralmente vem com `id = null`
    

Exemplo de requisição:

PUT /todos/10

Body:

{  
  "descricao": "Atualizar tarefa",  
  "concluido": true  
}

O JSON não contém o campo id.

Logo, o objeto criado pelo Jackson será:

todo.id == null

---

# 3. O Papel do `setId(id)`

Quando você faz:

todo.setId(id);

Você está sincronizando:

- O ID da URL
    
- Com o ID da entidade
    

Trecho exato:

todo.setId(id);

Isso garante que a entidade terá:

id = 10

---

# 4. O Que o Hibernate Faz Internamente

Supondo que o Service use:

repository.save(todo);

O Spring Data executa a seguinte lógica interna:

if (entityInformation.isNew(entity)) {  
    entityManager.persist(entity);  
} else {  
    entityManager.merge(entity);  
}

A decisão de `isNew(entity)` depende do ID.

Se ID for null:

- Hibernate chama persist()
    
- Executa INSERT
    

Se ID não for null:

- Hibernate chama merge()
    
- Executa UPDATE
    

---

# 5. O Que Acontece se NÃO Fizer `setId(id)`

Se você remover:

todo.setId(id);

O fluxo será:

1. JSON é convertido
    
2. id == null
    
3. save() entende que é entidade nova
    
4. Hibernate executa INSERT
    
5. Novo registro é criado
    

Você não atualiza.  
Você cria outro registro.

Esse é o erro comum.

---

# 6. O Que Acontece em Tempo de Execução (Fluxo Completo)

### Requisição PUT /todos/10

1. DispatcherServlet recebe requisição
    
2. Jackson converte JSON em objeto
    
3. Objeto vem com:
    
    - descricao preenchido
        
    - concluido preenchido
        
    - id == null
        
4. Controller executa:
    

todo.setId(id);

5. Agora a entidade tem:
    
    - id = 10
        
6. Service chama:
    

repository.save(todo);

7. Spring Data detecta:
    
    - id != null
        
8. Hibernate executa:
    

update tb_todo  
set descricao = ?, fl_concluido = ?  
where id = ?

# 7. Entendendo `merge()` em Profundidade

Quando `merge()` é chamado:

- Hibernate verifica se já existe entidade gerenciada com aquele ID
    
- Se não existir:
    
    - Ele faz SELECT para buscar
        
    - Se encontrar → atualiza
        
    - Se não encontrar → pode gerar exceção ou comportamento inesperado
        

Importante:

`merge()` não atualiza diretamente o objeto recebido.

Ele:

- Cria uma instância gerenciada
    
- Copia os valores
    
- Retorna a instância gerenciada
    

Isso é importante em cenários avançados.

---

# 8. Estado da Entidade

Estados no JPA:

1. Transient → nunca foi persistida
    
2. Managed → dentro do contexto
    
3. Detached → já foi persistida mas não está gerenciada
    
4. Removed → marcada para exclusão
    

Quando você recebe JSON:

- A entidade está Transient
    
- Mesmo que o ID exista
    

Quando chama save():

- Hibernate decide persist ou merge

# 9. Impacto Arquitetural

Essa abordagem funciona, mas não é a mais segura.

Problema:

Você está confiando no cliente para fornecer dados corretos.

Boa prática arquitetural seria:

1. Buscar a entidade no banco
    
2. Atualizar apenas os campos permitidos
    
3. Salvar a entidade gerenciada
    

Exemplo arquiteturalmente mais correto:

public TodoEntity atualizar(Integer id, TodoEntity dadosAtualizados) {  
    TodoEntity existente = repository.findById(id)  
        .orElseThrow();  
  
    existente.setDescricao(dadosAtualizados.getDescricao());  
    existente.setConcluido(dadosAtualizados.getConcluido());  
  
    return repository.save(existente);  
}

Isso evita:

- Atualizar registro inexistente
    
- Sobrescrever dados indevidamente
    
- Criar registros acidentalmente
