
# 1. Conceito Arquitetural

O Spring Boot segue, por padrão, o modelo de **arquitetura em camadas (Layered Architecture)**:

Camadas:

1. Controller → camada de entrada (HTTP)
    
2. Service → camada de regra de negócio
    
3. Repository → camada de acesso a dados
    
4. Entity → modelo persistente
    

Cada camada tem responsabilidade única.

Separação de responsabilidades:

- Controller → lida com HTTP
    
- Service → lida com regra de negócio
    
- Repository → lida com banco
    
- Entity → representa tabela
    

Isso reduz acoplamento e aumenta testabilidade.

---

# 2. Controller

## 2.1 Conceito

O Controller é a porta de entrada HTTP da aplicação.

Ele:

- Recebe requisições REST
    
- Converte JSON em objeto Java
    
- Delega processamento para o Service
    
- Retorna resposta HTTP
    

Ele NÃO deve conter regra de negócio.

---

## 2.2 Papel da Anotação `@RestController`

@RestController

### O que significa

É uma combinação de:

@Controller  
@ResponseBody

O Spring faz:

- Registra essa classe como Bean
    
- Mapeia endpoints HTTP
    
- Serializa retorno automaticamente para JSON
    

Internamente:

- Usa DispatcherServlet
    
- Usa HandlerMapping
    
- Usa HttpMessageConverters (Jackson)
    

---

## 2.3 `@RequestMapping("todos")`

Define o path base:

/todos

Trecho:

@RequestMapping("todos")

Impacto:

Organiza a API por recurso.

---

## 2.4 `@PostMapping`

@PostMapping

Mapeia:

POST /todos

O Spring registra esse método como handler HTTP.

---

## 2.5 `@RequestBody`

public TodoEntity salvar(@RequestBody TodoEntity todo)

### O que acontece internamente

1. Requisição chega com JSON
    
2. DispatcherServlet recebe
    
3. Jackson converte JSON → TodoEntity
    
4. Spring injeta no parâmetro
    

Exemplo JSON:

{  
  "descricao": "Estudar Spring",  
  "concluido": false  
}

O campo `id` será null.

---

## 2.6 Código do Controller

@RestController  
@RequestMapping("todos")  
public class TodoController {  
  
    private TodoService service;  
  
    public TodoController(TodoService service) {  
        this.service = service;  
    }  
  
    @PostMapping  
    public TodoEntity salvar(@RequestBody TodoEntity todo) {  
        return this.service.salvar(todo);  
    }  
}

# 3. Service

## 3.1 Conceito

Service representa a camada de regra de negócio.

Mesmo que hoje ele apenas delegue para o repository, arquiteturalmente ele existe para:

- Centralizar regras
    
- Validar dados
    
- Orquestrar múltiplos repositórios
    
- Controlar transações
    

Ele é o coração da aplicação.

---

## 3.2 Papel da Anotação `@Service`

@Service

O que o Spring faz:

- Detecta via component scan
    
- Registra como Bean no ApplicationContext
    
- Permite injeção via construtor
    

Impacto arquitetural:

- Define camada explícita
    
- Permite aplicar aspectos (AOP)
    
- Facilita testes
    

---

## 3.3 Injeção de Dependência

public TodoService(TodoRepository repository) {  
    this.repository = repository;  
}

O Spring:

1. Cria proxy do Repository
    
2. Cria instância de TodoService
    
3. Injeta o proxy no construtor
    

O Repository NÃO é uma implementação manual.

O Spring Data gera dinamicamente um proxy que implementa:

JpaRepository<TodoEntity, Integer>

---

## 3.4 Código do Service

@Service  
public class TodoService {  
  
    private TodoRepository repository;  
  
    public TodoService(TodoRepository repository) {  
        this.repository = repository;  
    }  
  
    public TodoEntity salvar(TodoEntity novoTodo) {  
        return repository.save(novoTodo);  
    }  
}

# 4. Fluxo Completo em Tempo de Execução

## 4.1 Quando a aplicação sobe

1. Spring inicia ApplicationContext
    
2. Faz component scan
    
3. Registra:
    
    - TodoController
        
    - TodoService
        
    - Proxy do TodoRepository
        
4. Hibernate sobe
    
5. Conexão com banco é criada
    

---

## 4.2 Quando chega uma requisição POST

Fluxo:

Cliente → Controller → Service → Repository → Hibernate → Banco

Etapas detalhadas:

1. DispatcherServlet intercepta requisição
    
2. Encontra método mapeado
    
3. Converte JSON para TodoEntity
    
4. Chama:
    

service.salvar(todo);

5. Service chama:
    

repository.save(novoTodo);

6. Spring Data decide:
    
    - id == null → INSERT
        
7. Hibernate gera SQL
    
8. Banco executa
    
9. ID é retornado
    
10. Entidade atualizada
    
11. Controller retorna JSON
    

---

# 5. Por Que Não Colocar Repository Direto no Controller?

Você poderia fazer:

@RestController  
public class TodoController {  
  
    private TodoRepository repository;  
  
    public TodoController(TodoRepository repository) {  
        this.repository = repository;  
    }  
}

Mas isso cria problemas arquiteturais.

---

## 5.1 Problema 1 — Acoplamento

Controller ficaria acoplado à camada de persistência.

Se mudar regra de negócio, você altera controller.

---

## 5.2 Problema 2 — Reuso

Se amanhã:

- Criar outro endpoint
    
- Criar integração externa
    
- Criar fila (Kafka)
    
- Criar batch
    

Você teria que duplicar regra.

Service centraliza.

---

## 5.3 Problema 3 — Testabilidade

Você pode testar:

- Service isoladamente
    
- Controller com MockMvc
    
- Repository separadamente
    

Sem Service, perde separação.

---

## 5.4 Problema 4 — Transações

Normalmente Service é onde se coloca:

@Transactional

A transação pertence à regra de negócio, não ao Controller.

---

# 6. Impacto Arquitetural Real

Controller:

- Camada externa
    
- Pode mudar (REST → GraphQL → gRPC)
    
- Não deve conter regra
    

Service:

- Camada central
    
- Independente de tecnologia de transporte
    
- Define regras
    

Repository:

- Camada interna
    
- Pode trocar banco sem afetar Controller
    

Isso permite:

- Evolução da aplicação
    
- Testes isolados
    
- Manutenção profissional
    

---

# 7. Resumo Estrutural do Fluxo

|Camada|Responsabilidade|
|---|---|
|Controller|HTTP|
|Service|Regra de negócio|
|Repository|Banco|
|Entity|Modelo persistente|

---

# 8. Conclusão Técnica

Você precisa do Service mesmo que ele apenas delegue porque:

- Define fronteira arquitetural
    
- Centraliza regras
    
- Controla transações
    
- Reduz acoplamento
    
- Permite crescimento da aplicação
    
- Mantém padrão de arquitetura backend profissional
    

Sem Service, você cria uma aplicação funcional.

Com Service, você cria uma aplicação arquiteturalmente correta.
