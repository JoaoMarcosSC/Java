# 1. Records como Modelo de Transporte (DTO)

## 1.1 Conceito de `record`

public record ErroCampo(String campo, String erro) { }

`record` é um tipo imutável introduzido no Java 16.

Internamente o compilador gera:

- Construtor
    
- Getters
    
- equals()
    
- hashCode()
    
- toString()
    

É ideal para DTO porque:

- Não representa entidade persistente
    
- Não precisa de mutabilidade
    
- É apenas transporte de dados
    

Trecho:

public record ErroCampo(String campo, String erro) { }

Impacto arquitetural:

- Evita expor Entity diretamente
    
- Desacopla camada HTTP da camada de domínio
    

---

# 2. Estrutura da Resposta de Erro

public record ErroResposta (int status, String mensagem, List<ErroCampo> erro)

Essa classe define um contrato padrão de erro da API.

Campos:

- status → código HTTP
    
- mensagem → descrição geral
    
- erro → lista de erros específicos de campo
    

Trecho:

public record ErroResposta (int status, String mensagem, List<ErroCampo> erro)

Isso cria um padrão REST consistente.

Exemplo de resposta:

{  
  "status": 422,  
  "mensagem": "Erro de validação.",  
  "erro": [  
    {  
      "campo": "nome",  
      "erro": "Campo Obrigatório"  
    }  
  ]  
}

---

## 2.1 Métodos Fábrica Estáticos

public static ErroResposta respostaPadrao(String mensagem){  
    return new ErroResposta(HttpStatus.BAD_REQUEST.value(), mensagem, List.of());  
}

public static ErroResposta conflito(String mensagem){  
    return new ErroResposta(HttpStatus.CONFLICT.value(), mensagem, List.of());  
}

Função:

- Centralizar criação de erros
    
- Evitar duplicação
    
- Padronizar código HTTP
    

Trechos:

return new ErroResposta(HttpStatus.BAD_REQUEST.value(), mensagem, List.of());

return new ErroResposta(HttpStatus.CONFLICT.value(), mensagem, List.of());

# 3. DTO com Validação

public record AutorDTO(  
        UUID ID,  
        @NotBlank(message = "Campo Obrigatório")  
        String nome,  
        @NotNull(message = "Campo Obrigatório")  
        LocalDate dataNascimento,  
        @NotBlank(message = "Campo Obrigatório")  
        String nacionalidade)

## 3.1 Papel das Anotações de Validação

- `@NotBlank` → campo não pode ser null nem string vazia
    
- `@NotNull` → campo não pode ser null
    

Essas anotações são do Bean Validation (Jakarta Validation).

Internamente o Spring usa:

- Hibernate Validator como implementação
    
- HandlerMethodArgumentResolver
    
- ValidationAutoConfiguration
    

---

# 4. O Papel do `@Valid` no Controller

Trecho:

@PostMapping    
public ResponseEntity<Void> salvar(@RequestBody @Valid AutorDTO autor)

## 4.1 O que o Spring faz internamente

Fluxo quando chega requisição:

1. DispatcherServlet intercepta
    
2. Jackson converte JSON → AutorDTO
    
3. Antes de chamar o método:
    
    - Spring executa validação
        
    - Usa Validator configurado
        
4. Se houver erro:
    
    - Lança `MethodArgumentNotValidException`
        
    - O método do controller NÃO é executado
        

Trecho crítico:

@RequestBody @Valid AutorDTO autor

---

# 5. Construção do ResponseEntity

Trecho:

URI location = ServletUriComponentsBuilder  
        .fromCurrentRequest()  
        .path("/{id}")  
        .buildAndExpand(autorEntidade.getId())  
        .toUri();  
  
return ResponseEntity.created(location).build();

## 5.1 O que acontece

1. `fromCurrentRequest()` pega URL atual
    
2. `path("/{id}")` adiciona sufixo
    
3. `buildAndExpand()` substitui o ID
    
4. `created(location)` retorna HTTP 201
    
5. Header Location é configurado
    

Resposta HTTP:

201 Created  
Location: /autores/{uuid}

Arquiteturalmente correto segundo REST.

`@RestControllerAdvice` é:

- - @ResponseBody implícito
        

Ele intercepta exceções lançadas nos controllers.

# 7. Tratamento de `MethodArgumentNotValidException`

@ExceptionHandler(MethodArgumentNotValidException.class)  
@ResponseStatus(HttpStatus.UNPROCESSABLE_ENTITY)

## 7.1 O que acontece quando a validação falha

1. @Valid detecta erro
    
2. Spring lança `MethodArgumentNotValidException`
    
3. DispatcherServlet procura handler
    
4. Encontra método anotado com @ExceptionHandler
    
5. Executa esse método
    
6. Retorna ErroResposta
    

Trecho:

@ExceptionHandler(MethodArgumentNotValidException.class)

---

# 8. Construção da Lista de Erros

List<FieldError> fieldErrors = e.getFieldErrors();

`FieldError` contém:

- nome do campo
    
- mensagem configurada
    

Conversão para ErroCampo:

List<ErroCampo> listaErros = fieldErrors  
        .stream()  
        .map(fe -> new ErroCampo(fe.getField(), fe.getDefaultMessage()))  
        .collect(Collectors.toList());

Isso cria uma estrutura desacoplada do Spring.

---

# 9. Resposta Final de Erro

return new ErroResposta(  
        HttpStatus.UNPROCESSABLE_ENTITY.value(),  
        "Erro de validação.",  
        listaErros);

Código HTTP:

422 — Unprocessable Entity

Isso significa:

- Requisição válida sintaticamente
    
- Mas semanticamente inválida
    

Arquiteturalmente correto para validação.

# 10. Fluxo Completo em Runtime

### Caso válido:

1. JSON recebido
    
2. Jackson converte
    
3. Validação passa
    
4. Controller executa
    
5. Service salva
    
6. Banco persiste
    
7. 201 Created é retornado
    

---

### Caso inválido:

1. JSON recebido
    
2. Jackson converte
    
3. Validação falha
    
4. Spring lança MethodArgumentNotValidException
    
5. GlobalExceptionController intercepta
    
6. Lista de erros é criada
    
7. 422 retornado com JSON estruturado
    

---

# 11. Impacto Arquitetural

Essa estrutura:

- Separa DTO de Entity
    
- Centraliza tratamento de erro
    
- Padroniza resposta REST
    
- Remove lógica de erro do controller
    
- Evita try/catch espalhado
    
- Permite crescimento da API
    

Em um backend real isso é obrigatório.
